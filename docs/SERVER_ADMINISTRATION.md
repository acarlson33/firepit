# Server Administration

## Scope

This document covers the administrative surface of Firepit: server management, roles, permissions, invite workflows, moderation, member access, and audit logging.

## Server Lifecycle

The main server APIs are:

- `GET /api/servers`
- `POST /api/servers/create`
- `GET /api/servers/public`
- `POST /api/servers/join`
- `GET /api/servers/{serverId}/stats`
- `GET /api/servers/{serverId}/members`
- `GET /api/servers/{serverId}/permissions`
- `POST /api/servers/{serverId}/mute`

Server stats and membership endpoints support server-level dashboards, moderation panels, and administrative summaries.

## Roles And Permission Overrides

Role management is exposed through:

- `GET|POST|PUT|DELETE /api/roles`
- `GET|POST|DELETE /api/role-assignments`
- `GET|POST|PUT|DELETE /api/channel-permissions`

The permission model combines:

- server ownership
- global admin or moderator teams
- server roles
- channel-level overrides

The application uses these permissions to decide whether a member can view channels, send messages, manage roles, administer a server, or perform moderation.

## Invite Management

Invite functionality is split between server-scoped management and public resolution:

- `GET|POST /api/servers/{serverId}/invites`
- `GET|DELETE /api/invites/{code}`
- `POST /api/invites/{code}/join`

Current invite behavior:

- invite codes are generated by the server-side invite library
- server owners and global admins can create and list invites
- revoke permissions are broader: server owner, invite creator, or global admin
- invite previews are public, but joining requires an authenticated user session
- invite records track `creatorId`, `currentUses`, expiry, maximum uses, and optional temporary semantics

## Moderation

Moderation actions are handled through `POST /api/servers/{serverId}/moderation`.

Supported actions:

- `ban`
- `unban`
- `mute`
- `unmute`
- `kick`

Permission checks allow moderation by:

- the server owner
- a member with effective server-management privileges
- a global moderator or admin

Moderation flows also enforce the following constraints:

- a moderator cannot act on themselves
- the server owner cannot be moderated through this endpoint
- mute and ban actions depend on the corresponding Appwrite collections being configured

## Audit Logging

Audit visibility is available through:

- `GET /api/servers/{serverId}/audit-logs`
- `GET /api/servers/{serverId}/audit-logs/export`

The audit route intentionally supports both current and legacy audit documents. Current documents may store denormalized moderation fields at the top level, while older records may only store equivalent values inside `meta` or `metadata`.

The API enriches audit responses with profile data when available and returns a normalized shape centered on:

- action
- moderator identity
- target user identity
- reason
- details
- timestamp

## Admin Panel Expectations

The admin experience should be able to:

- review server stats and member lists
- inspect effective permissions
- create and revoke invites
- assign and remove roles
- configure channel permission overrides
- mute, kick, ban, and unban members
- read and export audit history

## Relationship To Feature Flags

Audit logging behavior is feature-flag aware. If audit logging is disabled, moderation should continue to function, but audit-specific UI must degrade gracefully and avoid implying a complete history exists.

## Data Quality Notes

- Older audit records may require metadata fallback.
- Invite schema names in documentation should use `creatorId` and `currentUses`.
- Permission decisions should be described in terms of effective access, not a single role lookup.
